# 仓库协作说明

## 当前目标
当前工作的核心是梳理项目的推理流程，为后续基于 C++ 框架部署做准备。默认优先关注评估与推理链路，不主动展开训练细节、论文原理或模型设计分析，除非明确提出。

## 回答规则
1. 尽量使用中文。
2. 语言简洁，避免空话、客套话和明显的 AI 化表达。
3. 默认先给结论，再展开分析。

## 分析范围
优先查看以下链路：

- `tools/*.sh`：推理或评估入口脚本
- `eval.py`：推理主入口
- `engine.py`：输出解析、后处理、评估与可视化
- `datasets/`：输入图像与标注的加载方式
- `data_preprocess/`：点云到密度图的生成逻辑
- `models/roomformer.py`：模型实际输出张量的定义

如果问题与部署相关，重点说明：

- 模型输入的数据类型、shape、归一化方式
- 模型输出张量的含义、shape、阈值与筛选规则
- 后处理中的几何约束、类别分流和可视化是否属于部署必需步骤
- Python 依赖中哪些只是训练/评估工具，哪些属于推理必需

## 工作方式
处理问题时，默认按下面顺序：

1. 先确认入口脚本与实际执行路径。
2. 再定位输入、输出和中间后处理。
3. 最后补充相关代码位置、文件路径和必要的调用关系。

回答尽量直接给出：

- 结论
- 对应代码文件
- 关键函数或关键行
- 对部署是否有影响

## 修改约束
除非明确要求，不主动改动训练代码、模型结构和算法逻辑。若需要新增说明文档，优先写中文，内容围绕“推理流程、输入输出、后处理、部署关注点”组织。
